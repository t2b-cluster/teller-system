1. ภาพรวมระบบ Observability

ระบบ Observability ของ Teller System ถูกออกแบบตามหลัก Three Pillars of Observability ครอบคลุมทั้ง Metrics, Logs และ Traces พร้อม deploy บน GKE Autopilot ผ่าน Jenkins CI/CD Pipeline

📊 Metrics (Prometheus + prom-client)

ทุก service มี MetricsModule ในตัว expose /api/v1/metrics endpoint — Prometheus scrape ทุก 15 วินาที ครอบคลุม HTTP metrics, Business metrics (โอนเงิน, เปิดบัญชี, login) และ Node.js runtime metrics

📝 Logs (ELK Stack)

Filebeat DaemonSet เก็บ logs จากทุก pod → Elasticsearch แยก index ตาม service (teller-auth-*, teller-transfer-*, teller-system-*) → Kibana + Grafana สำหรับ search/visualize

🔗 Traces (Jaeger + OTel)

OpenTelemetry Collector รับ traces ผ่าน OTLP gRPC → export ไป Jaeger สำหรับ distributed tracing ข้าม service

📈 Dashboards (Grafana)

4 dashboards auto-provisioned: Infrastructure (Process CPU/RAM, Redis), Services (Request Rate, Latency, Status), Business Metrics (โอนเงิน, บัญชี, login), System Logs (ELK)

💡 Deployment: ระบบ Observability deploy บน GKE Autopilot ผ่าน Jenkins Pipeline อัตโนมัติ — local development ใช้ docker compose up -d

Data Flow

1
แต่ละ service มี MetricsModule ในตัว (src/metrics/) expose /api/v1/metrics — transfer-service และ transaction-service ใช้ @SetMetadata('isPublic', true) เพื่อ bypass JWT Guard
2
Prometheus scrape metrics ทุก 15 วินาที — ไม่ใส่ service label ใน scrape config เพื่อหลีกเลี่ยง exported_service prefix (ใช้ label จาก prom-client แทน)
3
Filebeat DaemonSet อ่าน logs จาก /var/log/containers/*.log (GKE Autopilot อนุญาตเฉพาะ /var/log/) → ส่งตรงไป Elasticsearch
4
Grafana query ข้อมูลจาก Prometheus (uid: prometheus), Elasticsearch (uid: elasticsearch), Jaeger (uid: jaeger) แสดงผลใน 4 dashboards
5
Business metrics ถูก increment ใน service code จริง: transfer.service.ts, account.service.ts, auth.service.ts

4. Metrics (Prometheus + prom-client)

4.1 MetricsModule (ทุก Service)

ทุก service มี src/metrics/ folder ประกอบด้วย 4 ไฟล์:

metrics.module.ts

Global NestJS module — auto-register default Node.js metrics ตอน onModuleInit

metrics.service.ts

สร้าง Prometheus Registry + Counter/Histogram/Gauge ทั้งหมด

metrics.controller.ts

GET /metrics endpoint — ใช้ @SetMetadata('isPublic', true) bypass JWT Guard

metrics.interceptor.ts

Auto-record http_requests_total + http_request_duration_seconds ทุก request

4.2 HTTP Metrics (อัตโนมัติ)

http_requests_total
จำนวน HTTP requests ทั้งหมด — increment อัตโนมัติผ่าน MetricsInterceptor
Labels: method, path, status, service
http_request_duration_seconds
Response time (histogram) — buckets: 10ms, 50ms, 100ms, 250ms, 500ms, 1s, 2.5s, 5s, 10s
Labels: method, path, status, service

4.3 Business Metrics (increment ใน service code)

teller_transfer_total
จำนวนการโอนเงิน — increment ใน transfer.service.ts
Labels: status (pending, failed) — pending เมื่อสร้าง transaction สำเร็จ, failed เมื่อ catch error
teller_transfer_amount_total
ยอดเงินโอนรวม — increment ด้วย .inc({ currency: 'THB' }, dto.amount)
Labels: currency
teller_account_created_total
จำนวนเปิดบัญชีใหม่ — increment ใน account.service.ts หลัง save สำเร็จ
teller_transactions_total
จำนวน transaction ทั้งหมด — increment ใน transfer.service.ts
Labels: status (pending, failed), type (TRANSFER)
teller_auth_login_total
จำนวน login — increment ใน auth.service.ts ทั้ง success และ failed
Labels: status (success, failed)
teller_reconciliation_total
ผลการ reconciliation
Labels: result (matched, mismatched)

4.4 Node.js Runtime Metrics (อัตโนมัติจาก prom-client)

# Stage: Deploy Observability Stack

  --from-file=observability/grafana/dashboards/ \
  -n tellersystem-uat --dry-run=client -o yaml | kubectl apply -f -

# 2. Apply k8s manifests
kubectl apply -f k8s/observability.yaml -n tellersystem-uat
kubectl apply -f k8s/observability-ingress.yaml -n tellersystem-uat

# 3. Wait for rollout
kubectl rollout status deployment/grafana -n tellersystem-uat

9.6 Business Metrics Instrumentation

// transfer.service.ts — หลัง transfer สำเร็จ
this.metrics.transferTotal.inc({ status: 'pending' });
this.metrics.transferAmountTotal.inc({ currency: 'THB' }, dto.amount);
this.metrics.transactionsTotal.inc({ status: 'pending', type: 'TRANSFER' });

// transfer.service.ts — เมื่อ transfer ล้มเหลว
this.metrics.transferTotal.inc({ status: 'failed' });
this.metrics.transactionsTotal.inc({ status: 'failed', type: 'TRANSFER' });

// account.service.ts — หลังเปิดบัญชีสำเร็จ
this.metrics.accountCreatedTotal.inc();

// auth.service.ts — login สำเร็จ/ล้มเหลว
this.metrics.authLoginTotal.inc({ status: 'success' });
this.metrics.authLoginTotal.inc({ status: 'failed' });

9.7 Useful PromQL Queries

# Request rate per service
sum(rate(http_requests_total[5m])) by (job)

# P95 latency per service
histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, job)) * 1000

# Transfer count by status
sum(teller_transfer_total) by (status)

# Total transfer amount
sum(teller_transfer_amount_total)

# Login success rate
sum(teller_auth_login_total{status="success"}) / sum(teller_auth_login_total) * 100
