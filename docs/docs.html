Sequence Diagram

sequenceDiagram participant T as Teller (Frontend) participant K as Kong Gateway participant TS as Transfer Service participant R as Redis participant DB as SQL Server participant Q as BullMQ T->>K: POST /transfers + Bearer Token + Idempotency Key K->>TS: Route to Transfer Service TS->>DB: Check Idempotency Key alt Key exists DB-->>TS: Cached response TS-->>T: Return cached result else Key not found TS->>R: Acquire Distributed Lock (Redlock) R-->>TS: Lock acquired TS->>DB: BEGIN TRANSACTION TS->>DB: SELECT account WITH LOCK (Pessimistic) DB-->>TS: Account balance alt Balance sufficient TS->>DB: UPDATE balance = balance - amount TS->>DB: INSERT transaction (PENDING) TS->>DB: INSERT outbox_message (PENDING) TS->>DB: COMMIT TS->>DB: Save Idempotency Key (24h TTL) TS->>Q: Publish transfer.initiated TS->>R: Release Lock TS-->>T: { transactionId, status: PENDING } else Insufficient balance TS->>DB: ROLLBACK TS->>R: Release Lock TS-->>T: 400 Bad Request end end

5. Logic ของ API

  1. ตรวจ Idempotency Key จาก DB (idempotency_keys table) — ถ้ามีอยู่แล้ว return cached response ทันที
  2. ตรวจว่า fromAccount ≠ toAccount
  3. Acquire Redis Distributed Lock (Redlock) บน lock:account:{fromAccount} TTL 5 วินาที
  4. เปิด Single DB Transaction:
    • Pessimistic Lock (SELECT ... WITH LOCK) บน accounts row
    • ตรวจยอดคงเหลือ ≥ amount
    • หักยอด (UPDATE accounts SET balance = balance - amount)
    • สร้าง transaction record (status = PENDING)
    • เขียน outbox_messages (status = PENDING) — Outbox Pattern
  5. บันทึก Idempotency Key ใน DB (TTL 24 ชั่วโมง)
  6. Publish message เข้า BullMQ (transfer-queue + notification-queue)
  7. Release lock

7. Functional / Non-Functional Requirements

Functional
  • FR-001 Transfer Money — โอนเงินผ่าน Core Banking + Idempotency Key
  • FR-002 Race Condition Handling — Pessimistic Lock + Distributed Lock
  • FR-004 Transaction Status Mgmt — PENDING → SUCCESS/FAILED via Outbox
Non-Functional
  • Performance: API Response ≤ 500ms
  • Data Integrity: ยอดถูกต้อง 100% แม้ Race Condition
  • Security: Bearer Token required, input validation

5. Logic ของ API

  1. สร้าง QueryBuilder จาก TypeORM
  2. Apply filters ตาม query params (accountId ค้นทั้ง from_account และ to_account)
  3. ใช้ Cursor-based Pagination (WHERE created_at < :cursor) แทน OFFSET — ประสิทธิภาพดีกว่ามากสำหรับ 50M+ records
  4. ORDER BY created_at DESC, TAKE limit+1 (เพื่อตรวจ hasMore)
  5. ถ้าได้ rows > limit → hasMore=true, pop row สุดท้าย, nextCursor = last item's createdAt

Sequence Diagram

sequenceDiagram participant T as Teller (Frontend) participant K as Kong Gateway participant AS as Auth Service participant DB as SQL Server participant BS as Business Service Note over T,AS: Login Flow T->>K: POST /auth/login { username, password } K->>AS: Route (Public) AS->>DB: SELECT user WHERE username AND isActive DB-->>AS: User record AS->>AS: bcrypt.compare(password, hash) AS->>AS: Sign JWT (30min) + Generate Refresh Token (7d) AS->>DB: INSERT refresh_tokens AS->>DB: UPDATE users SET lastLogin AS-->>T: { accessToken, refreshToken, expiresIn, user } T->>T: Store in sessionStorage Note over T,BS: API Call with Bearer Token T->>K: GET /balance/X + Authorization: Bearer JWT K->>BS: Route to Business Service BS->>AS: GET /auth/validate + Bearer JWT AS->>AS: jwt.verify(token) AS-->>BS: { valid: true, payload } BS-->>T: API Response Note over T,AS: Token Refresh (auto, 2min before expiry) T->>K: POST /auth/refresh { refreshToken } K->>AS: Route (Public) AS->>DB: Find refresh token (not revoked, not expired) AS->>DB: Revoke old token (rotation) AS->>AS: Sign new JWT + new Refresh Token AS-->>T: { accessToken, refreshToken }

5. Logic ของ API

  1. Login: ค้นหา user (username + isActive=true) → bcrypt.compare password → ออก JWT access token (30 นาที) + refresh token (random 64 bytes, 7 วัน) → บันทึก refresh_tokens table → อัพเดท lastLogin
  2. Refresh: ตรวจ refresh token ใน DB (not revoked, not expired) → revoke token เก่า (rotation) → ออก token ใหม่ทั้งคู่
  3. Logout: revoke refresh token ใน DB
  4. Validate: verify JWT signature → return payload (sub, username, role, branchCode)

JWT Payload: { sub: userId, username, role: TELLER|SUPERVISOR|ADMIN, branchCode }

7. Functional / Non-Functional Requirements

Functional
  • Teller ต้อง login ก่อนใช้งาน
  • Token หมดอายุ → redirect login
  • Role-based access (TELLER, SUPERVISOR, ADMIN)
Non-Functional
  • Security: bcrypt hash, JWT, refresh token rotation, OWASP compliance
  • Availability: Auth service แยก → ไม่กระทบ business services เมื่อ deploy

Sequence Diagram

sequenceDiagram participant OW as Outbox Worker (Cron 5s) participant DB as SQL Server participant CB as Core Banking participant Q as BullMQ participant RS as Reconcile (Cron 1min) Note over OW,CB: Outbox Worker OW->>DB: SELECT outbox WHERE status=PENDING DB-->>OW: Messages OW->>CB: Call Core Banking API alt Success CB-->>OW: OK OW->>DB: Credit account + status=SUCCESS + outbox=SENT OW->>Q: transfer.completed else Failed (max retries) OW->>DB: status=FAILED OW->>Q: transfer.failed end Note over RS,Q: Reconciliation RS->>DB: SELECT PENDING transactions older than 5min RS->>CB: Check status RS->>DB: INSERT reconciliation_log alt MISMATCH RS->>Q: notify.alert end

5. Logic ของ API

Outbox Worker (Cron ทุก 5 วินาที):
  1. อ่าน outbox_messages ที่ status=PENDING (ทีละ 10)
  2. เรียก Core Banking API
  3. สำเร็จ → credit บัญชีปลายทาง + update transaction status=SUCCESS + mark outbox SENT + notify queue
  4. ล้มเหลว → increment retryCount, ถ้า retryCount ≥ maxRetries → mark FAILED + update transaction FAILED + notify queue
Reconciliation (Cron ทุก 1 นาที):
  1. ค้นหา transactions ที่ PENDING นานเกิน 5 นาที
  2. ตรวจสถานะจาก Core Banking
  3. เปรียบเทียบ → สร้าง reconciliation_logs (MATCH/MISMATCH)
  4. ถ้า MISMATCH → ส่ง alert ผ่าน notification queue

7. FR / NFR

Functional
  • FR-004 Transaction Status Mgmt — PENDING, Reconciliation, Callback/Retry
Non-Functional
  • Data Integrity: ยอดถูกต้อง 100%
  • Availability: Retry with backoff, max retries

5. Logic / Design

Three Pillars of Observability:

  1. Metrics (Prometheus) — http_request_duration_seconds, transfer_total, db_pool_connections, redis_cache_total, outbox_pending_messages
  2. Logs (ELK Stack) — Structured JSON logs พร้อม correlationId, service name, action, masked sensitive data, traceId
  3. Traces (Distributed Tracing) — ติดตาม request ตั้งแต่ Frontend → Backend → Redis → SQL Server → Core Banking

Alert Rules: HighErrorRate (>5%), SlowResponse (p99 >2s), DBFailover, OutboxBacklog (>100), RedisDown, HighCPU (>85%), ReconcileMismatch

4. API Specification

MethodEndpointBody / Response
POST/api/v1/accountsBody: { accountNumber, accountName, initialDeposit?, currency? }
Response: { id, accountNumber, accountName, balance, currency, status }
GET/api/v1/accountsResponse: AccountResponse[]

Sequence Diagram

sequenceDiagram participant T as Teller (Frontend) participant K as Kong Gateway participant TXS as Transaction Service participant DB as SQL Server T->>K: POST /accounts { accountNumber, accountName, initialDeposit } K->>TXS: Route (Protected) TXS->>DB: SELECT WHERE account_number = ? alt Exists DB-->>TXS: Found TXS-->>T: 409 Conflict else New DB-->>TXS: null TXS->>DB: INSERT accounts (ACTIVE) TXS-->>T: { id, status: ACTIVE } end

5. Logic

  1. ตรวจว่าเลขบัญชีไม่ซ้ำ (findOne) → ถ้าซ้ำ throw ConflictException (409)
  2. สร้าง account record (status=ACTIVE, balance=initialDeposit หรือ 0)
  3. Return account data
