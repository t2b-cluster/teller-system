3. E2E Scenario Details

E2E-01: Race Condition — 2 Tellers โอนเงินบัญชีเดียวกันพร้อมกัน

รายการรายละเอียด
ข้อสอบQ1 Handling balance update race conditions
Preconditionสร้างบัญชี RACE_ACC ยอด 1,500 บาท
Actionส่ง 2 POST /transfers พร้อมกัน (concurrent) โอน 1,000 บาท × 2 ครั้ง จากบัญชีเดียวกัน
ExpectedTransfer แรกสำเร็จ, ที่สอง FAILED (insufficient balance) — ยอดคงเหลือ ≥ 0 (ไม่ติดลบ)
วัดผลGET /balance/RACE_ACC → balance ≥ 0
พิสูจน์Pessimistic Lock (SQL Server) + Redis Distributed Lock (Redlock) ป้องกัน double debit

E2E-02: Idempotency — ส่ง request ซ้ำด้วย key เดียวกัน

รายการรายละเอียด
ข้อสอบQ4 / Coding A Idempotent Transfer API
Actionส่ง POST /transfers 3 ครั้ง ด้วย x-idempotency-key เดียวกัน
Expectedทั้ง 3 ครั้งได้ transactionRef เดียวกัน ยอดถูกหักแค่ครั้งเดียว
วัดผลREF1 === REF2 === REF3
พิสูจน์Idempotency Key ถูกเก็บใน DB (idempotency_keys table) + Redis cache — request ซ้ำ return cached response

E2E-03: Outbox Pattern — ข้อมูลส่งถึง Core Banking

รายการรายละเอียด
ข้อสอบQ4 Outbox Pattern + Transaction Status Management
Actionโอนเงิน → ตรวจสถานะทันที (PENDING) → รอ 8 วินาที → ตรวจสถานะอีกครั้ง
Expectedสถานะเริ่มต้น = PENDING → หลัง outbox worker ทำงาน = SUCCESS
วัดผลGET /transfers/:id/status → status === "SUCCESS"
พิสูจน์Outbox Worker (Cron ทุก 5 วินาที) อ่าน outbox_messages → เรียก Core Banking → อัพเดท transaction status

E2E-04: Reconciliation — ตรวจจับ PENDING transactions

รายการรายละเอียด
ข้อสอบQ4 Channel Timeout → Reconciliation
Actionเรียก GET /reconciliation/logs
ExpectedEndpoint ตอบกลับ (service ทำงาน) — reconciliation cron ตรวจ PENDING ที่ค้าง > 5 นาที
วัดผลHTTP 200 + valid JSON response
พิสูจน์Reconciliation Service มี Cron ทุกนาที ตรวจ transactions ที่ PENDING นานเกินไป → สร้าง reconciliation_logs + ส่ง alert

E2E-05: Transaction History Performance — 2M+ records

รายการรายละเอียด
ข้อสอบQ3 / Coding B 50M+ records performance
Preconditionบัญชี 1234567890 มี 2,000,000 transactions ใน DB
ActionQuery GET /transactions?accountId=1234567890&limit=50 + ทดสอบ page 2 ด้วย cursor
ExpectedResponse time < 2000ms, items = 50, hasMore = true, nextCursor มีค่า, page 2 ได้ 50 items
วัดผล5 assertions: item count, hasMore, nextCursor, response time, page 2 count
พิสูจน์Cursor-based pagination (ไม่ใช่ OFFSET) + DB Index บน created_at + Virtualized List ใน Frontend

E2E-06: Redis Cache — Balance Inquiry

รายการรายละเอียด
ข้อสอบQ7 Redis strategy for high latency Core Banking
Actionเรียก GET /balance/1001000001 2 ครั้งติดกัน
Expectedครั้งที่ 2 → source = "cache" (ภายใน 30s TTL)
วัดผลsource === "cache"
พิสูจน์Cache-Aside pattern: ตรวจ Redis ก่อน → ถ้าไม่มี query DB แล้ว cache 30 วินาที → ครั้งถัดไปได้จาก cache ทันที

E2E-07: Authentication — Bearer Token Full Flow

รายการรายละเอียด
ข้อสอบQ5 Frontend security — Token handling
Action8 steps: Login → ใช้ token → ไม่มี token → token ผิด → refresh → reuse old refresh → logout → wrong password
Assertions 1. Login returns JWT (starts with "eyJ")
2. Valid token → HTTP 200
3. No token → HTTP 401
4. Bad token → HTTP 401
5. Refresh returns new JWT
6. Old refresh token → HTTP 401 (rotation)
7. Logout returns success message
8. Wrong password → HTTP 401
พิสูจน์JWT Bearer Token + Refresh Token Rotation + Revocation ทำงานครบ lifecycle

E2E-08: XSS Prevention

รายการรายละเอียด
ข้อสอบQ5 XSS Prevention
Actionส่ง <script>alert(1)</script> ในช่อง description ของ transfer
ExpectedAPI ยอมรับ request (ไม่ crash) — Frontend ใช้ DOMPurify sanitize ก่อนแสดงผล
พิสูจน์DOMPurify sanitize ทุก input + CSP Header ป้องกัน inline script + Nginx security headers

E2E-09: Health Check — Liveness & Readiness Probes

รายการรายละเอียด
ข้อสอบQ6 OpenShift Deployment — Probes
Actionเรียก /health/live และ /health/ready ของ transfer-service + health endpoints ของ service อื่น
ExpectedLiveness → "ok", Readiness → "connected" (ตรวจ DB), ทุก service ตอบ
พิสูจน์Kubernetes/OpenShift ใช้ probes เหล่านี้ตัดสินว่า Pod พร้อมรับ traffic หรือไม่

E2E-10: API Gateway Rate Limiting

รายการรายละเอียด
ข้อสอบInfrastructure
Actionส่ง request ผ่าน Kong แล้วตรวจ response headers
ExpectedKong ส่ง RateLimit headers กลับมา (X-RateLimit-Limit, X-RateLimit-Remaining)
พิสูจน์Kong rate-limiting plugin ทำงาน — จำกัด 100 req/min

E2E-11: Account Registration

รายการรายละเอียด
ข้อสอบFeature

4.2 รัน Automated E2E

bash e2e/run-all.sh

Script จะ:

  • Login อัตโนมัติเพื่อรับ Bearer token
  • รันทุก scenario (E2E-01 ถึง E2E-11)
  • ตรวจผลด้วย assertions (assert_eq, assert_contains, assert_lt)
  • แสดงผล PASS/FAIL แต่ละ assertion พร้อมสี
  • สรุปผลรวม: X passed, Y failed, Z total
  • Exit code 0 = ผ่านทั้งหมด, 1 = มี fail

6. สรุป

ระบบ Teller System ถูกทดสอบครอบคลุมทุกเงื่อนไขจากข้อสอบ Technical Lead Examination:

  • Q1 Race Condition → Pessimistic Lock + Distributed Lock ป้องกัน double debit (E2E-01)
  • Q2 Failover → Readiness probe ตรวจ DB, retry config, connection pool (E2E-09)
  • Q3 50M+ Performance → Cursor pagination < 2s สำหรับ 2M records (E2E-05)
  • Q4 Idempotency + Outbox + Reconcile → ป้องกันซ้ำ, PENDING→SUCCESS, ตรวจจับ mismatch (E2E-02, 03, 04)
  • Q5 Security → Bearer token full lifecycle + XSS prevention + CSP (E2E-07, 08)
  • Q6 Deployment → Liveness/Readiness probes ทำงาน (E2E-09)
  • Q7 Redis Cache → Cache-Aside 30s TTL, source indicator (E2E-06)
  • Coding A → Idempotent Transfer API with Outbox (E2E-02, 03)
  • Coding B → Virtualized List + Cursor Pagination (E2E-05)

ทุก test สามารถรันอัตโนมัติด้วย bash e2e/run-all.sh และวัดผล pass/fail ได้ชัดเจน
