1. ภาพรวมระบบ (System Overview)

Teller System เป็นระบบ Channel Application สำหรับพนักงานหน้าเคาน์เตอร์ธนาคาร (Teller) ใช้สำหรับทำธุรกรรมโอนเงิน ตรวจสอบยอดคงเหลือ ดูประวัติธุรกรรม และเปิดบัญชีใหม่ โดยเชื่อมต่อกับ Core Banking ผ่าน Outbox Pattern

3. ฟังก์ชันตามหน้าจอ (Screen Functions & API Mapping)

3.1 หน้า Login (LoginPage.tsx)

หน้าเข้าสู่ระบบสำหรับพนักงาน Teller รองรับ username/password authentication ผ่าน JWT

ActionAPI EndpointMethodServiceLogic
เข้าสู่ระบบ /api/v1/auth/login POST Auth Service รับ username/password → ค้นหา user จาก DB → bcrypt.compare() เทียบ password hash → สร้าง JWT access token (30 นาที) + refresh token (7 วัน) → เก็บ refresh token ใน DB → return tokens + user info
Refresh Token /api/v1/auth/refresh POST Auth Service รับ refresh token → validate จาก DB (ไม่ revoked + ไม่หมดอายุ) → revoke token เก่า (Token Rotation) → สร้าง token ใหม่ทั้งคู่ → Frontend auto-refresh ก่อนหมดอายุ 2 นาที
ออกจากระบบ /api/v1/auth/logout POST Auth Service Revoke refresh token ใน DB → ลบ session จาก sessionStorage

เทคนิคที่ใช้

  • bcrypt — Hash password ด้วย salt rounds 10 ป้องกัน rainbow table attack
  • JWT (JSON Web Token) — Stateless authentication, payload มี sub, username, role, branchCode
  • Refresh Token Rotation — ทุกครั้งที่ refresh จะ revoke token เก่าทันที ป้องกัน token reuse
  • sessionStorage — เก็บ token ฝั่ง client (ไม่ใช่ localStorage เพื่อความปลอดภัย — ปิด tab หาย)

3.2 หน้าโอนเงิน (TransferPage.tsx)

หน้าสำหรับพนักงาน Teller ทำรายการโอนเงินระหว่างบัญชี รองรับ Idempotency และป้องกัน Race Condition

ActionAPI EndpointMethodServiceLogic
โอนเงิน /api/v1/transfers POST Transfer Service 5 ขั้นตอนใน 1 request:
1. ตรวจ Idempotency Key จาก DB — ถ้ามีอยู่แล้ว return ผลลัพธ์เดิม
2. Acquire Distributed Lock (Redlock) บนบัญชีต้นทาง — ป้องกัน Race Condition
3. DB Transaction (Pessimistic Lock):
  - SELECT account WITH LOCK → ตรวจยอดเงิน
  - UPDATE balance (debit ต้นทาง)
  - INSERT transaction record (status: PENDING)
  - INSERT outbox message (same transaction)
4. บันทึก Idempotency Key + response ใน DB (TTL 24 ชม.)
5. Publish event ไป BullMQ (transfer-queue) สำหรับ async processing
ดูสถานะ /api/v1/transfers/:id/status GET Transfer Service Query transaction by ID → return status (PENDING/SUCCESS/FAILED)

เทคนิคที่ใช้

  • Distributed Lock (Redlock) — ใช้ Redis lock บน key lock:account:{accountNumber} ป้องกัน 2 request โอนเงินจากบัญชีเดียวกันพร้อมกัน (Race Condition)
  • Pessimistic Lock (SELECT FOR UPDATE) — Lock row ระดับ DB เพิ่มเติมจาก Distributed Lock เป็น defense-in-depth
  • Idempotency Key — Client ส่ง x-idempotency-key header (UUID) ทุกครั้ง ถ้าส่ง key ซ้ำจะได้ผลลัพธ์เดิม ไม่ทำรายการซ้ำ
  • Outbox Pattern — เขียน outbox message ใน DB transaction เดียวกับ debit เพื่อ guarantee consistency ระหว่าง DB กับ message queue
  • DOMPurify — Sanitize input ฝั่ง Frontend ก่อนส่ง API ป้องกัน XSS
  • UUID v4 — Generate idempotency key ฝั่ง Frontend ด้วย uuid library

3.3 หน้าประวัติธุรกรรม (TransactionHistoryPage.tsx)

ActionAPI EndpointMethodServiceLogic

3.4 หน้ายอดคงเหลือ (BalancePage.tsx)

ActionAPI EndpointMethodServiceLogic
ดูยอดคงเหลือ /api/v1/balance/:accountId GET Transaction Service Cache-Aside Pattern:
1. ตรวจ Redis cache key balance:{accountId}
2. ถ้ามี → return จาก cache (source: "cache")
3. ถ้าไม่มี → query จาก DB → เก็บ cache 30 วินาที → return (source: "database")

เทคนิคที่ใช้

  • Cache-Aside (Lazy Loading) — อ่าน cache ก่อน ถ้า miss ค่อย query DB แล้วเขียน cache กลับ
  • TTL 30 วินาที — Balance cache หมดอายุเร็วเพื่อให้ข้อมูลค่อนข้าง fresh
  • Response บอก source — API return source: "cache" หรือ "database" เพื่อ debug/monitor

4. Design Patterns & เทคนิคสำคัญ

4.1 Outbox Pattern (Transactional Outbox)

ใช้เพื่อ guarantee consistency ระหว่าง Database กับ Message Queue — เขียน outbox message ใน DB transaction เดียวกับ business logic

Transfer Service
→
DB Transaction (debit + outbox)
→
Outbox Worker (Cron 5s)
→
Core Banking API
→
Credit + Update Status

ไฟล์ที่เกี่ยวข้อง

  • transfer-service/src/transfer/transfer.service.ts — เขียน outbox ใน DB transaction
  • reconciliation-service/src/outbox/outbox.worker.ts — Cron job อ่าน outbox → เรียก Core Banking → credit ปลายทาง → update status
  • transfer-service/src/entities/outbox-message.entity.ts — Entity: id, aggregateType, eventType, payload, status, retryCount

Flow

  1. Transfer Service: debit บัญชีต้นทาง + สร้าง transaction (PENDING) + เขียน outbox message — ทั้งหมดใน 1 DB transaction
  2. Outbox Worker (ทุก 5 วินาที): อ่าน outbox ที่ status=PENDING → เรียก Core Banking API
  3. ถ้าสำเร็จ: credit บัญชีปลายทาง + update transaction status=SUCCESS + outbox status=SENT
  4. ถ้า fail: เพิ่ม retryCount ถ้าเกิน maxRetries (5) → status=FAILED + reverse debit

4.2 Idempotency Pattern

ป้องกันการทำรายการซ้ำเมื่อ client ส่ง request ซ้ำ (network retry, double click)

วิธีการทำงาน

  1. Client ส่ง x-idempotency-key header (UUID) ทุก request
  2. Server ตรวจ key จาก idempotency_keys table
  3. ถ้ามีอยู่แล้ว → return response เดิมที่เก็บไว้ (ไม่ทำรายการซ้ำ)
  4. ถ้าไม่มี → ทำรายการปกติ → บันทึก key + response ใน DB (TTL 24 ชม.)

ไฟล์ที่เกี่ยวข้อง

  • transfer-service/src/transfer/transfer.service.ts — ตรวจ + บันทึก idempotency key
  • transfer-service/src/entities/idempotency-key.entity.ts — Entity: idempotencyKey (PK), response (JSON), expiresAt
  • frontend/src/pages/TransferPage.tsx — Generate UUID ด้วย uuid v4

4.3 Distributed Lock (Redlock)

ป้องกัน Race Condition เมื่อ 2 request โอนเงินจากบัญชีเดียวกันพร้อมกัน

วิธีการทำงาน

  1. ก่อนทำ transfer → acquire lock บน Redis key lock:account:{accountNumber} (TTL 5 วินาที)
  2. ถ้า lock ได้ → ทำ DB transaction (pessimistic lock + debit)
  3. เสร็จแล้ว → release lock
  4. ถ้า lock ไม่ได้ → request ที่ 2 จะรอหรือ fail

ไฟล์ที่เกี่ยวข้อง

  • transfer-service/src/infrastructure/redis-lock.service.ts — Redlock implementation
  • transfer-service/src/transfer/transfer.service.ts — acquire/release lock รอบ transfer logic

4.4 JWT Authentication Flow

Client
→ Bearer Token →
Kong Gateway
→
Backend Service
→ HTTP call →
Auth Service /validate

วิธีการทำงาน

  1. ทุก backend service มี JwtAuthGuard เป็น global guard
  2. Guard ดึง Bearer token จาก Authorization header
  3. เรียก Auth Service GET /api/v1/auth/validate พร้อม token
  4. Auth Service verify JWT → return { valid: true, payload }
  5. Guard attach request.user สำหรับ downstream use
  6. Endpoint ที่มี @Public() decorator จะ skip guard

ไฟล์ที่เกี่ยวข้อง

  • shared/src/guards/jwt-auth.guard.ts — Guard implementation
  • auth-service/src/auth/auth.controller.ts — /validate endpoint
  • auth-service/src/auth/token.service.ts — JWT sign/verify
