Teller System 101

คู่มือเริ่มต้นพัฒนาสำหรับนักพัฒนาที่ยังไม่คุ้นเคยกับ Tech Stack นี้

1. โปรเจกต์นี้คืออะไร?

Teller System เป็น ระบบสำหรับพนักงานหน้าเคาน์เตอร์ธนาคาร (Teller) ใช้สำหรับ:

  • เข้าสู่ระบบด้วย username/password
  • โอนเงินระหว่างบัญชี
  • ดูยอดคงเหลือ
  • ดูประวัติธุรกรรม
  • เปิดบัญชีใหม่

ระบบถูกออกแบบเป็น Microservices — แทนที่จะเป็นโปรแกรมก้อนเดียว ถูกแบ่งเป็นหลาย service เล็กๆ แต่ละตัวทำหน้าที่เฉพาะ สื่อสารกันผ่าน HTTP API และ Message Queue

2. Tech Stack อธิบายทีละตัว

2.1 TypeScript

TypeScript คือ JavaScript ที่เพิ่ม ระบบ Type เข้ามา ช่วยจับ bug ตั้งแต่ตอนเขียนโค้ด ไม่ต้องรอรันแล้วค่อยเจอ error

// JavaScript ปกติ — ไม่รู้ว่า user มี field อะไร function greet(user) { return "Hello " + user.name; } // TypeScript — บอกชัดเจนว่า user ต้องมี name interface User { name: string; role: string; } function greet(user: User): string { return "Hello " + user.name; }
ทุกไฟล์ในโปรเจกต์นี้ลงท้ายด้วย .ts หรือ .tsx (TypeScript + React JSX)

2.2 NestJS (Backend Framework)

NestJS คือ framework สำหรับสร้าง backend API ด้วย TypeScript มีโครงสร้างชัดเจน คล้าย Angular ใช้ Decorator pattern (@Controller, @Get, @Post) ทำให้อ่านง่าย

// ตัวอย่าง Controller — กำหนด API endpoint @Controller('balance') // URL prefix: /balance export class BalanceController { @Get(':accountId') // GET /balance/1001000001 async getBalance(@Param('accountId') accountId: string) { return this.balanceService.getBalance(accountId); } }

โครงสร้างหลักของ NestJS

ส่วนไฟล์หน้าที่
Module*.module.tsรวม Controller + Service เข้าด้วยกัน เหมือน "กล่อง" ที่จัดกลุ่ม
Controller*.controller.tsรับ HTTP request → เรียก Service → return response
Service*.service.tsBusiness logic ทั้งหมด — คำนวณ, query DB, เรียก API อื่น
Entity*.entity.tsกำหนดโครงสร้างตาราง DB (mapping กับ TypeORM)
DTO*.dto.tsกำหนดรูปแบบ request body + validation rules
ทุก backend service ในโปรเจกต์นี้ใช้ NestJS ทั้งหมด (auth, transfer, transaction, reconciliation, notification)

2.3 React (Frontend)

React คือ library สำหรับสร้างหน้าเว็บ ใช้แนวคิด Component — แบ่งหน้าจอเป็นชิ้นเล็กๆ แต่ละชิ้นจัดการ state ของตัวเอง

// ตัวอย่าง Component — หน้า Login export default function LoginPage() { const [username, setUsername] = useState(''); // state เก็บค่า input const [password, setPassword] = useState(''); const handleSubmit = async (e) => { e.preventDefault(); await login(username, password); // เรียก API login }; return ( <form onSubmit={handleSubmit}> <input value={username} onChange={e => setUsername(e.target.value)} /> <input type="password" value={password} onChange={e => setPassword(e.target.value)} /> <button type="submit">เข้าสู่ระบบ</button> </form> ); }

ไฟล์สำคัญของ Frontend

ไฟล์หน้าที่
App.tsxComponent หลัก — ถ้ายังไม่ login แสดง LoginPage, ถ้า login แล้วแสดง sidebar + หน้าต่างๆ
auth/auth-context.tsxจัดการ login/logout/token — เก็บ state ของ user ที่ login อยู่
api/client.tsฟังก์ชันเรียก API — แนบ token อัตโนมัติ, จัดการ error 401
pages/*.tsxแต่ละหน้าจอ — Login, โอนเงิน, ประวัติ, ยอดคงเหลือ, เปิดบัญชี

2.4 MS SQL Server (Database)

ฐานข้อมูลหลักของระบบ เก็บข้อมูลบัญชี, ธุรกรรม, ผู้ใช้ ใช้ TypeORM เป็น ORM (Object-Relational Mapping) แทนที่จะเขียน SQL ตรงๆ

// แทนที่จะเขียน SQL: // SELECT * FROM accounts WHERE account_number = '1001000001' // ใช้ TypeORM แบบนี้: const account = await this.accountRepo.findOne({ where: { accountNumber: '1001000001' } });

Entity คืออะไร?

Entity คือ class ที่ map กับตาราง DB — แต่ละ property คือ column

@Entity('accounts') // map กับตาราง accounts export class Account { @PrimaryGeneratedColumn('uuid') // column id — auto-generate UUID id: string; @Column({ name: 'account_number' }) // column account_number accountNumber: string; @Column({ type: 'decimal' }) // column balance balance: number; }

2.5 Redis

Redis คือ in-memory database ที่เร็วมาก ในโปรเจกต์นี้ใช้ 3 อย่าง:

การใช้งานอธิบายตัวอย่าง
Cacheเก็บข้อมูลชั่วคราวเพื่อไม่ต้อง query DB ทุกครั้งยอดคงเหลือ cache 30 วินาที
Message Queueส่งงานให้ service อื่นทำแบบ async (ผ่าน BullMQ)โอนเงินเสร็จ → ส่ง notification
Distributed Lockป้องกัน 2 request ทำงานพร้อมกันบนข้อมูลเดียวกันLock บัญชีตอนโอนเงิน

2.6 Kong API Gateway

Kong เป็น "ประตูหน้า" ของระบบ — ทุก request จาก Frontend ต้องผ่าน Kong ก่อน Kong จะ route ไปยัง service ที่ถูกต้อง

Frontend
→ /api/v1/transfers →
Kong (:8000)
→
Transfer Service (:3001)

Kong ยังทำหน้าที่:

  • Rate Limiting — จำกัดจำนวน request ต่อนาที ป้องกัน DDoS
  • CORS — อนุญาตให้ Frontend เรียก API ข้าม domain
  • Logging — บันทึก request ทุกตัว

2.7 Docker & Kubernetes (GKE)

Docker — "บรรจุ" โปรแกรมพร้อม dependencies ทั้งหมดลงใน container ทำให้รันได้เหมือนกันทุกเครื่อง

Kubernetes (GKE) — จัดการ container หลายตัวบน cloud — auto-restart ถ้า crash, auto-scale ถ้า load สูง

3. Service สื่อสารกันอย่างไร

ระบบมี 2 วิธีหลักในการสื่อสารระหว่าง service:

วิธีที่ 1: HTTP API (Synchronous)

Service A เรียก Service B ตรงๆ แล้วรอ response — ใช้เมื่อต้องการผลลัพธ์ทันที

Transfer Service
→ HTTP GET →
Auth Service /validate
→ { valid: true } →
Transfer Service

ตัวอย่าง: ทุก service ตรวจ JWT token โดยเรียก Auth Service ผ่าน HTTP

วิธีที่ 2: Message Queue (Asynchronous)

Service A ส่ง "ข้อความ" ไปใน queue แล้วไม่ต้องรอ — Service B จะมาหยิบไปทำเมื่อพร้อม

Transfer Service
→ publish →
Redis Queue
→ consume →
Notification Service

ตัวอย่าง: โอนเงินเสร็จ → ส่ง event ไป notification-queue → Notification Service หยิบไปแจ้งเตือน

4. Walkthrough: เมื่อ Teller กดโอนเงิน เกิดอะไรขึ้น?

สมมติ Teller กดโอนเงิน 1,000 บาท จากบัญชี A ไปบัญชี B:

ขั้นตอนที่ 1: Frontend ส่ง Request

// TransferPage.tsx const idempotencyKey = uuidv4(); // สร้าง UUID เช่น "a1b2c3d4-..." api.transfer({ fromAccount: "1001000001", toAccount: "1001000002", amount: 1000, currency: "THB" }, idempotencyKey); // ส่ง HTTP POST ไปที่: // POST /api/v1/transfers // Headers: Authorization: Bearer eyJ..., x-idempotency-key: a1b2c3d4-...

ขั้นตอนที่ 2: Kong Gateway รับ Request

Kong ดู path /api/v1/transfers → route ไปที่ transfer-service:3001

ขั้นตอนที่ 3: JWT Guard ตรวจ Token

// jwt-auth.guard.ts // 1. ดึง token จาก header "Authorization: Bearer eyJ..." // 2. เรียก Auth Service: GET http://auth-service:3005/api/v1/auth/validate // 3. Auth Service verify JWT → return { valid: true, payload: { sub, username, role } } // 4. ถ้า valid → ปล่อยผ่าน, ถ้าไม่ → return 401

ขั้นตอนที่ 4: Transfer Controller รับ Request

// transfer.controller.ts @Post() async createTransfer( @Body() dto: CreateTransferDto, // { fromAccount, toAccount, amount, currency } @Headers('x-idempotency-key') idempotencyKey, // UUID จาก Frontend ) { return this.transferService.executeTransfer(dto, idempotencyKey); }

ขั้นตอนที่ 5: Transfer Service ทำงาน (หัวใจของระบบ)

// transfer.service.ts — executeTransfer() // 5.1 ตรวจ Idempotency — เคยทำรายการนี้แล้วหรือยัง? const existing = await this.idemRepo.findOne({ where: { idempotencyKey } }); if (existing) return JSON.parse(existing.response); // ถ้าเคย → return ผลเดิม // 5.2 Acquire Distributed Lock — ล็อคบัญชีต้นทาง const lock = await this.lockService.acquire(`lock:account:${dto.fromAccount}`, 5000); try { // 5.3 DB Transaction — ทำทุกอย่างใน 1 transaction (ถ้า fail จะ rollback ทั้งหมด) const result = await this.dataSource.transaction(async (manager) => { // 5.3.1 Pessimistic Lock — ล็อค row ของบัญชีใน DB const account = await manager.createQueryBuilder(Account, 'a') .setLock('pessimistic_write') .where('a.account_number = :num', { num: dto.fromAccount }) .getOne(); // 5.3.2 ตรวจยอดเงิน if (account.balance < dto.amount) throw new Error('Insufficient balance'); // 5.3.3 หักเงินบัญชีต้นทาง await manager.update(Account, ..., { balance: () => `balance - ${dto.amount}` }); // 5.3.4 สร้าง transaction record (status: PENDING) const tx = manager.create(Transaction, { status: 'PENDING', ... }); await manager.save(tx); // 5.3.5 เขียน Outbox Message (ใน transaction เดียวกัน!) const outbox = manager.create(OutboxMessage, { status: 'PENDING', ... }); await manager.save(outbox); return { transactionId: tx.id, status: 'PENDING' }; }); // 5.4 บันทึก Idempotency Key await this.idemRepo.save({ idempotencyKey, response: JSON.stringify(result) }); // 5.5 Publish event ไป Message Queue await this.transferProducer.publishTransferInitiated(result.transactionRef, dto); return result; // { transactionId, transactionRef, status: "PENDING" } } finally { await this.lockService.release(lock); // 5.6 ปล่อย Lock }

ขั้นตอนที่ 6: Outbox Worker ทำงาน (ทุก 5 วินาที)

// outbox.worker.ts — processOutbox() ทำงานทุก 5 วินาที // 6.1 อ่าน outbox messages ที่ status = PENDING const messages = await this.outboxRepo.find({ where: { status: 'PENDING' } }); for (const msg of messages) { // 6.2 เรียก Core Banking API (จำลอง) const result = await this.callCoreBanking(payload); if (result.success) { // 6.3 เติมเงินบัญชีปลายทาง await this.accountRepo.update(..., { balance: () => `balance + ${amount}` }); // 6.4 Update transaction status → SUCCESS await this.txRepo.update({ transactionRef }, { status: 'SUCCESS' }); // 6.5 Update outbox → SENT msg.status = 'SENT'; } // 6.6 ส่ง notification await this.notificationQueue.add('transfer.completed', { transactionRef, status: 'SUCCESS' }); }

ขั้นตอนที่ 7: Notification Service รับ Event

// notification.processor.ts async process(job: Job) { switch (job.name) { case 'transfer.completed': this.logger.log(`Transfer completed: ${job.data.transactionRef}`); // TODO: ส่งแจ้งเตือนไป Teller UI break; case 'transfer.failed': this.logger.warn(`Transfer failed: ${job.data.transactionRef}`); break; } }
สรุป: Frontend → Kong → Guard → Controller → Service (Lock → DB Transaction → Outbox) → Queue → Worker → Core Banking → Notification

5. วิธีเพิ่มฟีเจอร์ใหม่

5.1 เพิ่ม API Endpoint ใหม่ (Backend)

สมมติต้องการเพิ่ม API GET /api/v1/transfers/history ใน Transfer Service:

Step 1: เพิ่ม method ใน Service

// transfer.service.ts async getHistory(accountId: string) { return this.txRepo.find({ where: { fromAccount: accountId }, order: { createdAt: 'DESC' }, take: 50, }); }

// transfer.controller.ts @Get('history') async getHistory(@Query('accountId') accountId: string) { return this.transferService.getHistory(accountId); }

9. Pattern ที่ใช้บ่อย (อธิบายแบบง่าย)

9.1 Outbox Pattern — "เขียน DB กับส่ง Queue ต้องสำเร็จพร้อมกัน"

ปัญหา: ถ้าหักเงินใน DB สำเร็จ แต่ส่ง message ไป queue ไม่สำเร็จ → เงินหายแต่ไม่มีใครรู้

วิธีแก้: เขียน "outbox message" ลง DB ใน transaction เดียวกับการหักเงิน → มี worker มาอ่าน outbox แล้วส่งต่อ

ผลลัพธ์: ถ้า DB commit สำเร็จ → outbox message อยู่ใน DB แน่นอน → worker จะส่งต่อได้เสมอ

9.2 Idempotency — "กดซ้ำกี่ครั้งก็ได้ผลเหมือนกัน"

ปัญหา: User กดปุ่มโอนเงิน 2 ครั้ง (double click) หรือ network retry → เงินถูกโอน 2 ครั้ง

วิธีแก้: ทุก request ต้องมี "idempotency key" (UUID) ถ้า key ซ้ำ → return ผลลัพธ์เดิม ไม่ทำรายการซ้ำ

ผลลัพธ์: ไม่ว่าจะส่ง request กี่ครั้งด้วย key เดียวกัน เงินจะถูกโอนแค่ครั้งเดียว

9.3 Distributed Lock — "ห้ามทำพร้อมกัน"

ปัญหา: 2 Teller โอนเงินจากบัญชีเดียวกันพร้อมกัน → ยอดเงินผิด (Race Condition)

วิธีแก้: ก่อนโอนเงิน → "ล็อค" บัญชีใน Redis → ทำรายการ → ปล่อยล็อค คนที่ 2 ต้องรอ

ผลลัพธ์: ทำรายการทีละคน ยอดเงินถูกต้องเสมอ

9.4 Cache-Aside — "ถามของเก่าก่อน ถ้าไม่มีค่อยไปหาใหม่"

ปัญหา: ทุกครั้งที่ดูยอดคงเหลือต้อง query DB → ช้า ถ้ามีคนดูเยอะ

วิธีแก้: เก็บยอดคงเหลือใน Redis (เร็วกว่า DB 100 เท่า) → ถ้ามี return ทันที → ถ้าไม่มี query DB แล้วเก็บ cache 30 วินาที

ผลลัพธ์: request ส่วนใหญ่ไม่ต้องไปถึง DB เลย

9.5 JWT Authentication — "บัตรผ่านดิจิทัล"

วิธีทำงาน:

  1. Login สำเร็จ → server สร้าง "token" (ข้อความเข้ารหัส) ส่งให้ client
  2. ทุก request หลังจากนั้น client แนบ token ไปด้วย
  3. Server ตรวจ token → ถ้าถูกต้อง + ไม่หมดอายุ → ปล่อยผ่าน
  4. Token หมดอายุ (30 นาที) → ใช้ refresh token ขอ token ใหม่

ข้อดี: Server ไม่ต้องเก็บ session — แค่ verify token ก็รู้ว่าเป็นใคร (Stateless)

11. คำศัพท์ที่ควรรู้

คำความหมาย
Microservicesสถาปัตยกรรมที่แบ่งระบบเป็น service เล็กๆ แต่ละตัวทำหน้าที่เฉพาะ deploy แยกกันได้
API Gateway"ประตูหน้า" ที่รับ request ทั้งหมดแล้ว route ไปยัง service ที่ถูกต้อง
JWTJSON Web Token — ข้อความเข้ารหัสที่ใช้ยืนยันตัวตน
ORMObject-Relational Mapping — เขียน code แทน SQL ตรงๆ
EntityClass ที่ map กับตาราง DB
DTOData Transfer Object — กำหนดรูปแบบข้อมูลที่รับ/ส่ง
GuardMiddleware ที่ตรวจสอบ request ก่อนเข้า Controller (เช่น ตรวจ token)
Decorator@ ข้างหน้า function/class เช่น @Controller, @Get — เพิ่มความสามารถให้ code
Message Queueระบบส่งข้อความระหว่าง service แบบ async (ไม่ต้องรอ)
BullMQLibrary สำหรับ message queue ที่ใช้ Redis เป็น backend
Cron Jobงานที่รันอัตโนมัติตามเวลาที่กำหนด (เช่น ทุก 5 วินาที)
Pessimistic Lockล็อค row ใน DB ไม่ให้คนอื่นแก้ไขจนกว่า transaction จะเสร็จ
Optimistic Lockใช้ version number ตรวจว่าข้อมูลถูกแก้ไขระหว่างทางหรือไม่
Cacheเก็บข้อมูลชั่วคราวในที่เร็วกว่า (Redis) เพื่อลด load บน DB
TTLTime To Live — ระยะเวลาที่ cache/token มีอายุ
Container"กล่อง" ที่บรรจุโปรแกรม + dependencies ทั้งหมด รันได้เหมือนกันทุกเครื่อง
Kubernetes (K8s)ระบบจัดการ container หลายตัว — auto-restart, auto-scale, load balance
HelmPackage manager สำหรับ Kubernetes (เหมือน npm แต่สำหรับ K8s)
CI/CDContinuous Integration / Continuous Deployment — build + test + deploy อัตโนมัติ
Pipelineลำดับขั้นตอนที่ทำอัตโนมัติ (build → test → deploy)
