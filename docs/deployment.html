2.2 Pipeline Detail — แต่ละ Stage

#StageToolรายละเอียดFail Criteria
1 Code Push GitHub Developer push code ไปยัง GitHub repository — trigger Jenkins webhook —
2 Build & Lint npm npm ci install dependencies, npm run lint ตรวจ ESLint rules, tsc compile TypeScript Lint error หรือ compile error → pipeline fail
3 GitLeaks Scan GitLeak สแกนหา secrets, API keys, passwords ที่หลุดเข้า source code พบ secret → pipeline fail ทันที
4 Vulnerability & Container Scan Trivy สแกน Docker image หา CVE vulnerabilities ใน OS packages และ npm dependencies Critical/High CVE → pipeline fail
5 Code Quality & Test Coverage SonarQube วิเคราะห์ code quality (bugs, code smells, duplications) + ตรวจ test coverage ≥ 80% Quality Gate failed หรือ coverage < 80% → pipeline fail
6 E2E Automate Test Playwright รัน E2E tests ครอบคลุม critical paths: Login → Transfer → Transaction History → Balance E2E test fail → pipeline fail
7 Image Push Harbor Registry Build Docker image แต่ละ service → tag ด้วย build number → push ไป Harbor (private registry) Push fail → pipeline fail
8 Deploy to OpenShift Helm + OpenShift Helm upgrade --install แต่ละ service → Blue-Green deployment → Health check Health check fail → auto rollback
9 OWASP ZAP Scan OWASP ZAP Dynamic Application Security Testing (DAST) — สแกน running application หา security vulnerabilities High/Critical finding → แจ้ง Security Team

2.3 Jenkinsfile

pipeline { agent { label 'openshift-agent' } environment { REGISTRY = 'harbor.internal.bank.com' IMAGE_PREFIX = 'teller-system' HELM_CHART = './helm/teller-system' SONAR_HOST = 'https://sonarqube.internal.bank.com' } stages { stage('Checkout') { steps { checkout scm } } stage('Install & Lint') { steps { sh 'npm ci' sh 'npm run lint' } } stage('GitLeaks Scan') { steps { sh 'gitleaks detect --source . --report-format json --report-path gitleaks-report.json' } } stage('Unit Test + Coverage') { steps { sh 'npm run test:cov -- --ci --reporters=default --reporters=jest-junit' } post { always { junit '**/coverage/junit.xml' publishHTML(target: [reportDir: 'coverage/lcov-report', reportFiles: 'index.html', reportName: 'Coverage']) } } } stage('SonarQube Analysis') { steps { withSonarQubeEnv('SonarQube') { sh """ sonar-scanner \ -Dsonar.projectKey=teller-system \ -Dsonar.sources=services/ \ -Dsonar.tests=services/ \ -Dsonar.test.inclusions=**/*.spec.ts \ -Dsonar.javascript.lcov.reportPaths=**/coverage/lcov.info \ -Dsonar.qualitygate.wait=true """ } } } stage('Docker Build & Trivy Scan') { steps { script { def services = ['auth-service','transfer-service','transaction-service', 'reconciliation-service','notification-service','frontend'] for (svc in services) { sh "docker build -t ${REGISTRY}/${IMAGE_PREFIX}/${svc}:${BUILD_TAG} -f services/${svc}/Dockerfile ." sh "trivy image --severity HIGH,CRITICAL --exit-code 1 ${REGISTRY}/${IMAGE_PREFIX}/${svc}:${BUILD_TAG}" sh "docker push ${REGISTRY}/${IMAGE_PREFIX}/${svc}:${BUILD_TAG}" } } } } stage('E2E Test (Playwright)') { steps { sh 'npx playwright test --reporter=html' } post { always { publishHTML(target: [reportDir: 'playwright-report', reportFiles: 'index.html', reportName: 'E2E']) } } } stage('Deploy Staging') { steps { sh """ helm upgrade --install teller-staging ${HELM_CHART} \ -n staging \ --set global.image.tag=${BUILD_TAG} \ --set global.image.registry=${REGISTRY}/${IMAGE_PREFIX} \ -f ${HELM_CHART}/values-staging.yaml \ --wait --timeout 5m """ } } stage('OWASP ZAP Scan') { steps { sh """ zap-cli quick-scan --self-contained \ --start-options '-config api.disablekey=true' \ https://teller-staging.internal.bank.com """ } post { always { publishHTML(target: [reportDir: '.', reportFiles: 'zap-report.html', reportName: 'OWASP ZAP']) } } } stage('Deploy Production') { when { branch 'main' } steps { input message: 'Deploy to Production?', submitter: 'tech-lead,devops' sh """ helm upgrade --install teller-prod ${HELM_CHART} \ -n production \ --set global.image.tag=${BUILD_TAG} \ --set global.image.registry=${REGISTRY}/${IMAGE_PREFIX} \ -f ${HELM_CHART}/values-production.yaml \ --wait --timeout 10m """ } } } post { failure { slackSend(channel: '#teller-alerts', color: 'danger', message: "BUILD FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER} - ${env.BUILD_URL}") } success { slackSend(channel: '#teller-deploys', color: 'good', message: "Deployed ${BUILD_TAG} to ${env.BRANCH_NAME}") } } }

3. OpenShift Deployment Strategy

3.1 Blue-Green Deployment

เลือกใช้ Blue-Green Deployment ผ่าน OpenShift Route เพราะเป็นระบบธนาคารที่ต้องการ Zero Downtime อย่างเคร่งครัด โดยเฉพาะในช่วง Business Hours ที่ Teller กำลังให้บริการลูกค้า

Blue (Current)
← Traffic
|
Green (New)
← Smoke Test
  1. Deploy เวอร์ชันใหม่เป็น Green โดยไม่รับ Traffic
  2. รัน Smoke Test + Health Check บน Green
  3. สลับ OpenShift Route ให้ชี้ไปที่ Green (ใช้เวลา < 1 วินาที)
  4. เก็บ Blue ไว้เป็น Rollback — หากมีปัญหาสลับกลับทันที
  5. หลังยืนยันสำเร็จ (30 นาที) ลบ Blue เก่าออก
Business Hours Deploy: การ Deploy ในช่วงเวลาทำการต้องมั่นใจว่า Teller ทุกคนไม่ได้รับผลกระทบ ดังนั้น maxUnavailable: 0 เสมอ และมี Manual Approval ก่อน Production Deploy

ProbeEndpointตรวจอะไรConfig
Liveness GET /health/live Process ยังทำงานอยู่หรือไม่ — ถ้า fail Kubernetes จะ restart Pod initialDelay: 15s, period: 10s, failureThreshold: 3

4. Security Scanning Strategy

Pipeline มี Security Scan 4 ชั้น ครอบคลุมตั้งแต่ source code จนถึง running application

ชั้นToolประเภทสแกนอะไรเมื่อไหร่
1 GitLeaks Secret Detection API keys, passwords, tokens ที่หลุดเข้า source code ทุก commit (Stage 3)
2 SonarQube SAST (Static) Bugs, code smells, security hotspots, test coverage ทุก build (Stage 5)
3 Trivy Container Scan CVE vulnerabilities ใน OS packages, npm dependencies, Docker image ทุก Docker build (Stage 4)
4 OWASP ZAP DAST (Dynamic) XSS, SQL Injection, CSRF, security misconfig บน running application หลัง deploy staging (Stage 9)

5. Docker Image Management (Harbor)

5.1 Image Naming Convention

harbor.internal.bank.com/teller-system/{service-name}:{build-tag} # ตัวอย่าง harbor.internal.bank.com/teller-system/auth-service:build-142 harbor.internal.bank.com/teller-system/transfer-service:build-142 harbor.internal.bank.com/teller-system/transaction-service:build-142 harbor.internal.bank.com/teller-system/reconciliation-service:build-142 harbor.internal.bank.com/teller-system/notification-service:build-142 harbor.internal.bank.com/teller-system/frontend:build-142

5.2 Image Lifecycle

  • ทุก image ถูก scan ด้วย Trivy ก่อน push ไป Harbor
  • Harbor มี vulnerability scanning ซ้ำอีกชั้น
  • Image retention policy: เก็บ 30 builds ล่าสุด, ลบ image ที่เก่ากว่า 90 วัน
  • Production image ถูก tag เพิ่มเป็น :latest และ :v1.x.x (semantic version)

6. Rollback Strategy

สถานการณ์วิธี Rollbackเวลา
Health check fail หลัง deploy Helm auto-rollback (--wait --timeout) — Kubernetes จะ rollback อัตโนมัติ < 1 นาที
Bug พบหลัง deploy สำเร็จ Blue-Green: สลับ OpenShift Route กลับไป Blue (เวอร์ชันเก่า) < 1 วินาที
ต้อง rollback ไป version เก่ากว่า helm rollback teller-prod {revision} < 2 นาที
Database migration ผิดพลาด รัน reverse migration script + rollback Helm 5-10 นาที
